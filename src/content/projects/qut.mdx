Qut is a private AI coding lab running on Hephaestus — a self-hosted inference server, a dual-agent development workflow, and a network topology designed so that source code and AI queries never leave the local network. The name comes from Old Turkic: qut is the divine life-force that Tengri grants to those capable of wielding it. Not a god, not a servant — the intelligence itself.

The project exists because the alternative didn't survive contact with reality. [MoltBot](/projects/moltbot) was the original plan: a multi-channel AI agent orchestration system built on the OpenClaw runtime, with skill-based architecture, hybrid memory, and sandboxed tool execution. The design was sound. The ecosystem was not. In January 2026, Koi Security disclosed 341+ malicious skills in ClawHub (OpenClaw's public skill repository) — backdoors, credential exfiltrators, and prompt injection payloads disguised as productivity tools. Separately, Shodan scans found 1,009+ OpenClaw gateways exposed to the public internet with no authentication. The skill system that was supposed to be a feature had become a supply chain attack surface.

Rather than hardening a platform built on a compromised ecosystem, Qut takes the opposite approach: no third-party agent runtime, no skill marketplace, no gateway exposed to anything. Just a local model on a local GPU, talked to by local clients, on an air-gapped network segment. The intelligence is sovereign — it runs on your iron and answers to no one else's cloud.

### At a glance

- **Inference server:** llama.cpp on CUDA VM (192.168.100.100), Tesla P40 24GB, native Anthropic + OpenAI endpoints
- **Model:** Qwen3.5-35B-A3B, Unsloth Dynamic 2.0 UD-Q4_K_XL (20.6GB), 96K context, q4_0 KV cache, ~41 t/s
- **Chat UI:** Open WebUI (Docker, localhost-only)
- **Local agentic coding:** claude-local (Claude Code against local model, `CLAUDE_CONFIG_DIR` isolated) + Aider
- **Cloud agent:** Claude Code (paid Anthropic API, full agentic capability, final reviewer)
- **Network:** vmbr2 (192.168.100.0/24), air-gapped by default, egress opt-in via bastion NAT
- **Clients:** Any vmbr2 VM (direct) or Mac (via SSH tunnel through bastion)
- **Client setup:** Automated scripts for lab VMs (`setup-vmbr2.sh`) and external machines (`setup-external.sh`)

### Architecture

```
┌─────────────────────────────────────────────────────────┐
│  CUDA VM (192.168.100.100) — INFERENCE SERVER ONLY      │
│                                                         │
│  llama-server :8080 ──► Tesla P40 (Qwen3.5-35B-A3B)    │
│    ├── /v1/messages        (Anthropic, for claude-local)│
│    └── /v1/chat/completions (OpenAI, for aider-local)   │
│  Open WebUI   :3000 ──► chat UI (Docker)                │
│                                                         │
│  No project files. No Claude Code. Just inference.      │
│  Both auto-start on boot (systemd + Docker).            │
└──────────────────────────┬──────────────────────────────┘
                           │ vmbr2 (192.168.100.0/24)
            ┌──────────────┼──────────────┐
            │              │              │
     ┌──────┴──────┐ ┌────┴────┐  ┌──────┴──────┐
     │ kali  :20   │ │victim:30│  │ any VM ...  │
     │ claude-local │ │claude-l │  │ claude-l    │
     │ aider-local │ │aider-l  │  │ aider-l     │
     │ ↕ files     │ │↕ files  │  │ ↕ files     │
     └─────────────┘ └─────────┘  └─────────────┘
            │
     ┌──────┴──────────────────────────────────────┐
     │ Mac (via SSH tunnel through bastion)         │
     │ claude + claude-local + aider-local          │
     └─────────────────────────────────────────────┘
```

The CUDA VM is the brain — it loads the model into GPU memory and serves both Anthropic and OpenAI-compatible REST APIs natively from a single llama-server process. No translation proxy needed. It has no project files, no credentials, no coding tools. If it were somehow compromised, there's nothing useful on it. Defense in depth.

Client VMs are where the work happens. They hold the project files, run the coding agents, and send prompts to the CUDA VM over the internal network. The Mac connects via SSH tunnel through the bastion — prompts and completions travel the tunnel, but source code never leaves the Mac.

### The three-agent workflow

The real payoff of the setup: three AI frontends, each with different strengths, all sharing the same project files and git repo.

**claude-local (Lead Architect / Creative Director):** Claude Code talking to the local Qwen3.5 model. Long context, natural agent loops, unlimited usage, fully private. The one you open first for planning, brainstorming, and the "beautiful demo feel." Free and local.

**aider-local (Senior Engineer):** Aider talking to the same Qwen3.5 model. Fast execution, transparent edits, auto-commits to git, architect mode for plan-then-execute workflows. The one you give clear tasks and it delivers. Free and local.

**claude (Principal / Final Reviewer):** Paid Anthropic Claude Code. Full agentic capability with the strongest reasoning model available. You escalate to it for the hard problems, final reviews, or when the local model starts looping. Metered but unmatched.

You are the Project Manager — you decide who works on what and when. A typical session: ask claude-local to plan a feature and save it to PLAN.md. Hand the plan to aider-local to implement. When it commits, ask claude-local to review. If something is subtle or wrong, escalate to paid claude for a final opinion.

**Why three?** Paid Claude alone is powerful but metered — you wouldn't burn expensive API calls on routine implementation. The local model alone is fast and free but weaker on complex reasoning. claude-local and aider-local serve different interaction patterns (conversational agent vs. structured editor). All three together: best of all worlds.

### Network isolation

Security comes from network topology, not firewall rules.

VMs on vmbr2 (192.168.100.0/24) have no internet access by default. Not because of a firewall blocking them — there simply is no route. Their network interface is on an isolated L2 domain with no gateway configured. It's like plugging a computer into a switch that isn't connected to a router.

```
Internet ←→ [Router] ←→ vmbr0 ←→ [Bastion] ←→ vmbr2 ←→ Lab VMs
                                       ↑
                                 NAT + ip_forward
                                 (off by default)
```

When a VM needs internet (to install packages, download a model), you explicitly enable it:

```bash
sudo egress-on    # adds bastion as gateway, enables NAT
# ... do your downloads ...
sudo egress-off   # removes the gateway, back to isolated
```

During normal operation — when you're writing code with AI assistance — nothing touches the internet. The CUDA VM serves the model on 192.168.100.0/24. Client VMs send prompts on 192.168.100.0/24. There's no API key to leak, no data to exfiltrate, no cloud provider logging your queries.

The bastion is the single choke point. All external access flows through it: outbound only when egress is active, inbound SSH via ProxyJump. Your Mac can't even reach the lab VMs directly — they're on a different subnet that your apartment network doesn't route to.

### Why not MoltBot

[MoltBot](/projects/moltbot) was designed as a multi-channel AI agent orchestration system on the OpenClaw runtime. The architecture was solid: gateway control plane, file-governed planning, hybrid memory, skill layering, per-agent sandboxing. But the ecosystem it depended on had fundamental supply-chain problems:

- **341+ malicious skills** found in ClawHub (OpenClaw's public skill repository) by Koi Security in January 2026 — backdoors, credential exfiltrators, prompt injection payloads
- **1,009+ exposed gateways** found via Shodan with no authentication — "auth mode none" was the default until a recent breaking change
- **Skill trust model** was opt-out, not opt-in — new skills from the marketplace were executable by default unless explicitly denied

The lesson: if your agent runtime treats third-party skills as a feature rather than a supply chain, you inherit the security posture of every skill author. Auditing tools exist (skillvet, Cisco Skill Scanner), but they're reactive — you're always one step behind the next malicious upload.

Qut sidesteps the problem entirely. There is no skill marketplace. There is no gateway. The inference server listens on an air-gapped network segment. The coding agents (Claude Code, Aider) are the only tools, and they operate on local files with local models. The attack surface is the attack surface of SSH and your own code — which is exactly what you'd have without any AI at all.

### Quick reference

```bash
# Start SSH tunnel (Mac, one-time per session)
ssh -fN cuda

# Agentic coding with local model — fast executor (free, unlimited)
cd ~/my-project && aider-local

# Agentic coding with local model — Claude Code interface (free, unlimited)
cd ~/my-project && claude-local

# Agentic coding with paid Claude (powerful, metered)
cd ~/my-project && claude

# Chat with local model in browser
open http://localhost:3000

# Check GPU / model status
ssh cuda 'nvidia-smi'
ssh cuda 'curl -s localhost:8080/v1/models | python3 -m json.tool'

# Restart inference server
ssh cuda 'sudo systemctl restart llama-server'
```
