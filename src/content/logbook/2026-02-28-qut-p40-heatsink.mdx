Installing the aftermarket heatsink on the Tesla P40 after the thermal crash documented in the previous session, validating thermals under sustained inference, and tuning context from 64k/q8_0 to 96k/q4_0. The P40 is a passive-cooled 250W datacenter GPU running in a consumer tower — sustained inference load caused a PCIe fatal error and full GPU disappearance from the bus.

**Test 1: Short burst (2048 tokens, 64k/q8_0 config)**

| Phase | Temp | Power | Util |
|-------|------|-------|------|
| Idle  | 29°C | 51W   | 0%   |
| Peak  | 43°C | 143W  | 97%  |

41.4 tok/s. No issues.

This is the exact workload that caused the PCIe crash in the previous session — multi-turn agentic code generation from Mac via SSH tunnel.

### Context tuning: 64k/q8_0 to 96k/q4_0

64k context was thermally safe but too small for claude-code's agentic workflow. A single pytest run (7899 lines of output) exceeded the context window:

```
API Error: 400 {"error":{"code":400,"message":"request (71747 tokens) exceeds the
available context size (65536 tokens)","type":"exceed_context_size_error"}}
```

Claude-code accumulates system prompt + conversation history + tool call outputs. Multi-step tasks with test output regularly blow past 64k.

**The squeeze on 24 GB VRAM:**

| Context | KV cache (q8_0) | KV cache (q4_0) | Total VRAM | Fits P40? |
|---------|-----------------|-----------------|------------|-----------|
| 32k     | ~1.2 GB         | ~0.6 GB         | ~20.3 GB   | Yes       |
| 64k     | ~1.8 GB         | ~0.9 GB         | ~20.6 GB   | Yes       |
| **96k** | **~2.7 GB**     | **~1.4 GB**     | **~21.1 GB** | **Yes (q4_0 only)** |
| 128k    | ~3.6 GB         | ~1.8 GB         | ~21.5 GB   | Marginal  |
| 128k    | ~4.5 GB (q8_0)  | —               | ~24.2 GB   | Spills    |

**Fix:** increase context to 96k while halving KV cache precision to q4_0. The Qwen3.5-35B-A3B hybrid architecture (only 10/40 layers use standard attention) means KV cache is already small — q4_0 precision loss is diluted across the 30 linear-attention layers that don't use KV cache at all.

```diff
- -c 65536 --cache-type-k q8_0 --cache-type-v q8_0
+ -c 98304 --cache-type-k q4_0 --cache-type-v q4_0
```

| | Before (64k/q8_0) | After (96k/q4_0) |
|---|---|---|
| **Context window** | 65,536 tokens | **98,304 tokens** (+50%) |
| **KV precision** | q8_0 | q4_0 |
| **VRAM used** | 21.5 GB | **21.2 GB** (less!) |
| **VRAM free** | 2.9 GB | **3.2 GB** |
| **Inference speed** | 41.4 tok/s | **40.7 tok/s** (~same) |
| **Peak temp** | 43°C (short test) | 46°C (short test) |

50% more context, slightly less VRAM, virtually identical speed. Post-tuning `claude-local` session confirmed q4_0 KV doesn't meaningfully degrade agentic code quality for this model.

### Auth conflict fix (recurring)

The `~/.claude-local` config dir accumulated a cached `userID` from OAuth, causing:
```
Auth conflict: Both a token (claude.ai) and an API key (ANTHROPIC_API_KEY) are set.
```

Fix: `rm -rf ~/.claude-local && mkdir -p ~/.claude-local` — fresh config dir. The `userID` field in `.claude.json` is what triggers the conflict. If it returns, nuke the config dir.
