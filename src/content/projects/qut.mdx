Qut is a private AI lab running on Hephaestus — a self-hosted inference server, a three-agent coding workflow, an autonomous research assistant, and a knowledge graph that accumulates everything the system learns. The name comes from Old Turkic: qut is the divine life-force that Tengri grants to those capable of wielding it. Not a god, not a servant — the intelligence itself.

The project started as a reaction to [MoltBot](/projects/moltbot)'s failure — the OpenClaw ecosystem's supply-chain problems (341+ malicious ClawHub skills, 1,009+ exposed gateways) made third-party agent runtimes untenable. Qut took the opposite approach: local model, local GPU, air-gapped network. But it didn't stop at coding agents. In March 2026, the inference stack grew an autonomous layer: a Discord-resident assistant (Hermes) that observes activity, classifies what's interesting, searches the web, accumulates findings in a knowledge graph, and delivers daily research digests. The system now operates continuously — researching, learning, and reporting — whether or not anyone is talking to it.

### At a glance

- **Inference server:** llama.cpp on CUDA VM (192.168.100.100), Tesla P40 24GB, native Anthropic + OpenAI endpoints
- **Model:** Qwen3.5-35B-A3B, Unsloth Dynamic 2.0 UD-Q4_K_XL (20.6GB), 96K context, q4_0 KV cache, ~41 t/s
- **Chat UI:** Open WebUI (Docker, localhost-only)
- **Local agentic coding:** claude-local (Claude Code against local model, `CLAUDE_CONFIG_DIR` isolated) + Aider
- **Cloud agent:** Claude Code (paid Anthropic API, full agentic capability, final reviewer)
- **Network:** vmbr2 (192.168.100.0/24), air-gapped by default, egress opt-in via bastion NAT
- **Clients:** Any vmbr2 VM (direct) or Mac (via SSH tunnel through bastion)
- **Client setup:** Automated scripts for lab VMs (`setup-vmbr2.sh`) and external machines (`setup-external.sh`)

### Architecture

```
┌─────────────────────────────────────────────────────────┐
│  CUDA VM (192.168.100.100) — INFERENCE SERVER ONLY      │
│                                                         │
│  llama-server :8080 ──► Tesla P40 (Qwen3.5-35B-A3B)    │
│    ├── /v1/messages        (Anthropic, for claude-local)│
│    └── /v1/chat/completions (OpenAI, for aider-local)   │
│  Open WebUI   :3000 ──► chat UI (Docker)                │
│                                                         │
│  No project files. No Claude Code. Just inference.      │
│  Both auto-start on boot (systemd + Docker).            │
└──────────────────────────┬──────────────────────────────┘
                           │ vmbr2 (192.168.100.0/24)
            ┌──────────────┼──────────────┐
            │              │              │
     ┌──────┴──────┐ ┌────┴────┐  ┌──────┴──────┐
     │ kali  :20   │ │victim:30│  │ any VM ...  │
     │ claude-local │ │claude-l │  │ claude-l    │
     │ aider-local │ │aider-l  │  │ aider-l     │
     │ ↕ files     │ │↕ files  │  │ ↕ files     │
     └─────────────┘ └─────────┘  └─────────────┘
            │
     ┌──────┴──────────────────────────────────────┐
     │ Mac (via SSH tunnel through bastion)         │
     │ claude + claude-local + aider-local          │
     └─────────────────────────────────────────────┘
```

The CUDA VM is the brain — it loads the model into GPU memory and serves both Anthropic and OpenAI-compatible REST APIs natively from a single llama-server process. No translation proxy needed. It has no project files, no credentials, no coding tools. If it were somehow compromised, there's nothing useful on it. Defense in depth.

Client VMs are where the work happens. They hold the project files, run the coding agents, and send prompts to the CUDA VM over the internal network. The Mac connects via SSH tunnel through the bastion — prompts and completions travel the tunnel, but source code never leaves the Mac.

### The three-agent workflow

The real payoff of the setup: three AI frontends, each with different strengths, all sharing the same project files and git repo.

**claude-local (Lead Architect / Creative Director):** Claude Code talking to the local Qwen3.5 model. Long context, natural agent loops, unlimited usage, fully private. The one you open first for planning, brainstorming, and the "beautiful demo feel." Free and local.

**aider-local (Senior Engineer):** Aider talking to the same Qwen3.5 model. Fast execution, transparent edits, auto-commits to git, architect mode for plan-then-execute workflows. The one you give clear tasks and it delivers. Free and local.

**claude (Principal / Final Reviewer):** Paid Anthropic Claude Code. Full agentic capability with the strongest reasoning model available. You escalate to it for the hard problems, final reviews, or when the local model starts looping. Metered but unmatched.

You are the Project Manager — you decide who works on what and when. A typical session: ask claude-local to plan a feature and save it to PLAN.md. Hand the plan to aider-local to implement. When it commits, ask claude-local to review. If something is subtle or wrong, escalate to paid claude for a final opinion.

**Why three?** Paid Claude alone is powerful but metered — you wouldn't burn expensive API calls on routine implementation. The local model alone is fast and free but weaker on complex reasoning. claude-local and aider-local serve different interaction patterns (conversational agent vs. structured editor). All three together: best of all worlds.

### Network isolation

Security comes from network topology, not firewall rules.

VMs on vmbr2 (192.168.100.0/24) have no internet access by default. Not because of a firewall blocking them — there simply is no route. Their network interface is on an isolated L2 domain with no gateway configured. It's like plugging a computer into a switch that isn't connected to a router.

```
Internet ←→ [Router] ←→ vmbr0 ←→ [Bastion] ←→ vmbr2 ←→ Lab VMs
                                       ↑
                                 NAT + ip_forward
                                 (off by default)
```

When a VM needs internet (to install packages, download a model), you explicitly enable it:

```bash
sudo egress-on    # adds bastion as gateway, enables NAT
# ... do your downloads ...
sudo egress-off   # removes the gateway, back to isolated
```

During normal operation — when you're writing code with AI assistance — nothing touches the internet. The CUDA VM serves the model on 192.168.100.0/24. Client VMs send prompts on 192.168.100.0/24. There's no API key to leak, no data to exfiltrate, no cloud provider logging your queries.

The bastion is the single choke point. All external access flows through it: outbound only when egress is active, inbound SSH via ProxyJump. Your Mac can't even reach the lab VMs directly — they're on a different subnet that your apartment network doesn't route to.

### Why not MoltBot

[MoltBot](/projects/moltbot) was designed as a multi-channel AI agent orchestration system on the OpenClaw runtime. The architecture was solid: gateway control plane, file-governed planning, hybrid memory, skill layering, per-agent sandboxing. But the ecosystem it depended on had fundamental supply-chain problems:

- **341+ malicious skills** found in ClawHub (OpenClaw's public skill repository) by Koi Security in January 2026 — backdoors, credential exfiltrators, prompt injection payloads
- **1,009+ exposed gateways** found via Shodan with no authentication — "auth mode none" was the default until a recent breaking change
- **Skill trust model** was opt-out, not opt-in — new skills from the marketplace were executable by default unless explicitly denied

The lesson: if your agent runtime treats third-party skills as a feature rather than a supply chain, you inherit the security posture of every skill author. Auditing tools exist (skillvet, Cisco Skill Scanner), but they're reactive — you're always one step behind the next malicious upload.

Qut sidesteps the problem entirely. There is no skill marketplace. There is no gateway. The inference server listens on an air-gapped network segment. The coding agents (Claude Code, Aider) are the only tools, and they operate on local files with local models. The attack surface is the attack surface of SSH and your own code — which is exactly what you'd have without any AI at all.

### Autonomous assistant

In March 2026, the inference stack grew beyond coding agents into an autonomous research assistant. The system observes what the user is doing, decides what's interesting, searches the web, accumulates knowledge, and delivers daily digests — all running unattended on the same P40 that serves the coding workflow.

**Hermes** is the unified conversational and scheduling layer. It replaced OpenClaw's fragmented launchd scheduling with 11 cron jobs, manages 27+ skills (including the Sophia curiosity engine and autoresearch loop), and owns Discord interaction via the "Honda Civic HB 1992" bot on the QUT server.

**Observation pipeline.** Every 15 minutes, `observe.sh` runs 8 collectors (active app, browser URL, terminal commands with intent classification, git status, recent files, CUDA status, system metrics, ActivityWatch data). A rule-based classifier tags each observation as ESCALATE/DIGEST/STORE/IGNORE — replacing a failed attempt at sub-1B local LLM classification. IGNORE observations are skipped without touching CUDA.

**Knowledge graph.** `knowledge.db` stores everything the system learns: 82k+ entity nodes, 143k+ edges, typed relationships (mentions, related_to, found_via, answers, derived_from). Entity extraction runs hourly via Qwen NER. Pattern detection finds co-occurring entities. Hypothesis generation identifies structural gaps worth researching.

<div class="diagram-section"><a href="/artifacts/qut/knowledge-graph-obsidian.png" class="lightbox"><img src="/artifacts/qut/knowledge-graph-obsidian.png" alt="Obsidian graph view of the knowledge graph — 82k+ entity nodes rendered as a dense galaxy of interconnected clusters" loading="lazy" /></a></div>

**Sophia curiosity engine.** Three-signal scoring: novelty (how new relative to the graph), mastery (is the system making progress on this topic), coherence (does it connect to the user's active projects). Dynamic weighting shifts with user activity — deep work sessions increase coherence weight, idle time increases curiosity weight.

**Dual scoring model.** identity_score ("who you are" — durable, provenance-weighted, with floors so 3-year interests never decay) and activity_score ("what's live now" — 7-day exponential decay). A 3-year OpenAI ChatGPT export (1,084 conversations) provides the identity foundation.

**Obsidian vault.** The knowledge graph exports to `~/Obsidian/Hephaestus/` as 85k+ markdown files with wikilinks and YAML frontmatter. Daily digest notes create a chronological browsing surface. Both identity and activity scores are visible in frontmatter.

**Autoresearch.** Automated ML training experiments (forked Karpathy nanochat) time-share the P40 with inference — stop llama-server, train for 5-7 minutes, restart, ingest results. Key finding: FP32 is 7.5x faster than FP16 on Pascal (no tensor cores).

### Quick reference

```bash
# Start SSH tunnel (Mac, one-time per session)
ssh -fN cuda

# Agentic coding with local model — fast executor (free, unlimited)
cd ~/my-project && aider-local

# Agentic coding with local model — Claude Code interface (free, unlimited)
cd ~/my-project && claude-local

# Agentic coding with paid Claude (powerful, metered)
cd ~/my-project && claude

# Chat with local model in browser
open http://localhost:3000

# Check GPU / model status
ssh cuda 'nvidia-smi'
ssh cuda 'curl -s localhost:8080/v1/models | python3 -m json.tool'

# Restart inference server
ssh cuda 'sudo systemctl restart llama-server'
```

- **Feb 22** — Initial build. Qwen3-Coder-30B-A3B on P40, LiteLLM proxy, 32k context. Claude Code chat-only (tool-calling broken through LiteLLM), Aider pivot for agentic coding.
- **Feb 26** — Upgraded to Qwen3.5-35B-A3B (hybrid Mamba2/attention architecture). Rebuilt llama.cpp for native Anthropic endpoint, removed LiteLLM entirely. Context tuned to 64k (128k caused KV cache spill to RAM). Created automated client setup scripts. Fixed auth conflict with `CLAUDE_CONFIG_DIR` isolation — three-agent coexistence confirmed. P40 thermal crash during sustained load; heatsink ordered. Added q8_0 KV cache precision and single-slot optimization.
- **Feb 28** — Aftermarket heatsink installed on P40. Ziptie chaining with steel bar reinforcement for clamping pressure. Bottom mellanox fan flipped to feed air directly into GPU heatsink. Angled exhaust fan added. Thermal validation: 62°C peak under sustained load (32°C headroom to throttle), GPU safe for continuous use. Context tuned from 64k/q8_0 to 96k/q4_0 — 50% more context, less VRAM, same speed. Swapped to Unsloth Dynamic 2.0 UD-Q4_K_XL quantization for improved tool-calling and coding quality.
- **Mar 01-10** — OpenClaw Discord bot setup. Persistent Discord-resident assistant ("Honda Civic HB 1992" on the QUT server) connected to Qwen via autossh tunnel. Fought OpenClaw's model config system (3 API type attempts, Ollama discovery hook wiping config, deprecated format). Stabilized SSH tunnel with ControlMaster bypass. Discovered infinite retry loop bug — switched to direct `openclaw message send` for all automated delivery.
- **Mar 10-15** — Hybrid Autonomy (12 phases). Built the full autonomous pipeline: queue.db (SQLite WAL, 6 tables), 7-collector observation system (15-min cycle), rule-based classifier (ESCALATE/DIGEST/STORE/IGNORE — replaced failed sub-1B local LLM attempt), CUDA interpretation dispatch, 3 hunt lanes (tech-scout/paper-radar/security-watch), 12 Discord commands, digest generation, weekly backup. Two-mode architecture: Mode A (CUDA offline, queue everything) / Mode B (CUDA online, process everything).
- **Mar 11** — Autoresearch. Forked Karpathy's nanochat for automated ML training on the P40, time-sharing VRAM with inference. Ported to Pascal (SDPA replacing FA3, torch.compile disabled). Discovered FP32 is 7.5x faster than FP16 on the P40 — Pascal has no tensor cores, so FP16 runs on the same ALUs with conversion overhead.
- **Mar 12** — Context compaction fix. Lowered OpenClaw's contextWindow from 96k to 65k (actual llama-server still at 96k) to trigger compaction before hitting the real limit. Fixed UTF-8 crash from Japanese YouTube titles in observations.
- **Mar 15-18** — Search, curiosity, and knowledge graph. Replaced Brave API (HTTP 402) with self-hosted SearXNG. Built knowledge.db (nodes, edges, searches, events) with flock serialization. Hourly entity extraction via Qwen NER. Sophia curiosity engine: 3-signal scoring (novelty/mastery/coherence) with dynamic weighting based on user activity. Retired single-signal curiosity.sh.
- **Mar 18** — Hermes consolidation. Replaced fragmented launchd + OpenClaw scheduling with Hermes as unified conversational and scheduling layer. 10 scripts retired, 8+ launchd plists replaced by 11 Hermes cron jobs. Fixed 119 backlogged findings (6 days undelivered). Obsidian vault export — knowledge graph rendered as 540 browsable markdown files with wikilinks and daily notes at ~/Obsidian/Hephaestus/.
- **Mar 25** — Interest signal pipeline. Dual scoring model: identity_score ("who you are," durable, provenance-weighted) + activity_score ("what's live now," 7-day decay). Imported 3-year OpenAI ChatGPT history (1,084 conversations, 471 new nodes). Interest inbox at ~/interests/inbox/ for importing PDFs, markdown, etc.
- **Mar 29** — Entity dedup. Knowledge graph had grown to 86,874 entity nodes with thousands of duplicates. Phase 1: deterministic normalization (3,969 merged in 4 seconds). Phase 2: token blocking + RapidFuzz + Qwen LLM judgment (861 merged across 48,200 candidate pairs, ~3.5h GPU time, 4 crash-restarts over residential WiFi). Final graph: 82,105 entities, 143,050 edges, 0 false merges. Obsidian vault re-exported to 84,036 files.
