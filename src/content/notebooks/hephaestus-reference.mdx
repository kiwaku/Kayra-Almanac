## Network Topology

Three bridges, strict isolation by default.

| Bridge | Physical NIC | Purpose | Host IP |
|--------|-------------|---------|---------|
| **vmbr0** | nic0 | Apartment LAN / WAN uplink | 10.13.235.35/24 |
| **vmbr1** | nic1 (I350 port1) | Physical lab switch (GS105E), L2 only | none |
| **vmbr2** | none | Internal VM-to-VM, no internet | 192.168.100.1/24 |

## Egress Control

Default state: **no ambient internet** on vmbr2 VMs. Egress is explicitly toggled.

### Enable egress (two-step)

```bash
# Step 1: On bastion — enable NAT
sudo egress-on
# Enables ip_forward, iptables MASQUERADE for 192.168.100.0/24 via ens18

# Step 2: On CUDA VM — add route + DNS
sudo egress-on
# Adds default route via 192.168.100.10, sets DNS to 1.1.1.1 / 8.8.8.8
```

### Disable egress (reverse order)

```bash
# Step 1: On CUDA VM
sudo egress-off
# Removes default route, restores original DNS

# Step 2: On bastion
sudo egress-off
# Drops iptables rules, disables ip_forward
```

### Remote one-liners (from bastion)

```bash
# Enable
sudo egress-on && ssh -t kudakuda@192.168.100.100 'sudo /usr/local/sbin/egress-on'

# Disable
ssh -t kudakuda@192.168.100.100 'sudo /usr/local/sbin/egress-off' && sudo egress-off
```

### Verification (on CUDA VM)

```bash
ip route | head                   # Should show/not show default route
ping -c 1 1.1.1.1                # Test connectivity
getent hosts huggingface.co       # Test DNS
```

## SSH Access

### From Mac (external)

```bash
ssh bastion                       # Jump host
ssh cuda                          # CUDA VM (via ProxyJump through bastion)
ssh -fN cuda                      # Background tunnel for llama-server access
```

SSH config uses `ProxyJump bastion` for all vmbr2 hosts. Mac cannot reach 192.168.100.0/24 directly — different subnet, no route.

### Key gotchas

- **ClearAllForwardings:** when SSHing to CUDA for commands (not tunneling), use `-o ClearAllForwardings=yes` to avoid port conflicts with active autossh tunnels
- **ControlMaster conflict:** long-lived tunnels (autossh) must use `-o ControlPath=none -o ControlMaster=no` to avoid stale mux socket death
- **AUTOSSH_GATETIME=0:** required for launchd-started autossh (network may not be ready at boot)

## InfiniBand Fabric

Dual ConnectX-3 Pro (MT27520, fw 2.38.5000) with QSFP+ DAC internal loop.

### Port wiring

**Port 1 to port 1, port 2 to port 2.** Cross-wiring causes "link detected: no."

### Interface mapping

```
HCA #1 (native slot, 0000:03:00.0)
  ens1   -> port 0
  ens1d1 -> port 1

HCA #2 (M.2 riser, 0000:09:00.0)
  enp9s0   -> port 0
  enp9s0d1 -> port 1
```

### Bandwidth reality

- HCA #1 (Gen3 x8): full 40 Gb/s capable
- HCA #2 (Gen2 x4): caps at ~11-12 Gb/s (PCIe bottleneck, not DAC)
- Good for: stability testing, RDMA plumbing, burn-in, 10-20G class work
- Not for: true 40GbE line-rate validation

### IPoIB subnets

- `10.0.40.0/24` (link 1)
- `10.0.41.0/24` (link 2)

### Driver state

```bash
# Check drivers
lsmod | grep mlx4                 # mlx4_en (Ethernet), mlx4_ib, ib_core, ib_uverbs
ibv_devinfo                       # IB device info, link layer, port state
ibstat                            # Quick port status
opensm                            # Subnet manager (runs on host, HCA #1)
```

## P40 Thermal

Aftermarket heatsink with ziptie+steel bar clamping. Bottom Mellanox fan flipped to feed air into GPU heatsink. Angled exhaust fan added.

- **Peak under sustained load:** 62°C (32°C headroom to throttle at 94°C)
- **24V blower:** powered via 12V->24V step-up converter on dedicated Molex. **Never connect blower directly to PSU.**

## Hardware Debugging

### BIOS

- Board: ZX-DU99D4 V1.12
- PCIe mode: **Throughput Mode** (Slot Map A) — required for current layout
- Memory: DDR4 ECC RDIMM, requires specific slot population order

### Recovery procedures

- **BIOS reflash:** CH341A programmer + SOIC-8 clip. Flash from known-good dump.
- **I350-T4 NVM recovery:** eeupdate utility to reflash MAC addresses and NVM image after soft-brick. Required Linux boot environment with the I350 visible to the kernel.
- **Memory init failure (DU99D4):** specific DIMM slot population order matters. RTFM for this board.

### Useful diagnostics

```bash
lspci -t                          # PCIe topology tree
lspci -vvv -s 03:00.0            # Detailed device info (link speed, width)
dmesg | grep -i pci               # PCIe enumeration messages
dmesg | grep -i iommu             # IOMMU grouping
cat /proc/interrupts              # IRQ distribution
sensors                           # Thermal sensors (if lm-sensors configured)
```
